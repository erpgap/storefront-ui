# Cache Invalidation

## Summary

Storefront UI uses Redis-backed Nitro storage for two cache layers:

- **Page cache**: rendered HTML generated by `cachedEventHandler` in `server/plugins/route-cache.ts`
- **GraphQL query cache**: Odoo GraphQL responses generated by `cachedFunction` in `server/api/odoo/query.post.ts`

When product, category, or homepage content changes in Odoo, both layers should be invalidated. Clearing only the page cache can still regenerate HTML from stale GraphQL data.

## Endpoints

| Endpoint | Use when | What it clears |
|---|---|---|
| `POST /api/internal/cache/invalidate/product` | A product page changed | Product page HTML + product template data + product variant data |
| `POST /api/internal/cache/invalidate/category` | A category page changed | Category page HTML + category data + product list data |
| `POST /api/internal/cache/invalidate/homepage` | Homepage content changed | Homepage HTML + homepage data |
| `POST /api/internal/invalidate-cache` | Legacy/autodetect endpoint | Detects page type for known prefixed paths |

Prefer the type-specific endpoints. Storefront UI receives product/category slugs from the Odoo GraphQL base configured in `NUXT_PUBLIC_ODOO_BASE_URL`, for example `/product/the-zen-platform-bed-47`, so the type-specific endpoints are more reliable than autodetection.

## Authentication

Every request must include:

| Header | Description |
|---|---|
| `x-invalidation-key` | Static shared key configured in Nuxt |
| `x-invalidation-timestamp` | Unix time in milliseconds |
| `x-invalidation-signature` | HMAC-SHA256 request signature |

The signature payload is:

```text
METHOD:PATH:TIMESTAMP:CANONICAL_JSON
```

Example:

```text
POST:/api/internal/cache/invalidate/product:1710000000000:{"dryRun":true,"url":"/product/the-zen-platform-bed-47"}
```

`CANONICAL_JSON` must sort object keys alphabetically at every nesting level and must not contain whitespace.

## Environment Variables

Set these in the Nuxt runtime environment:

| Variable | Description |
|---|---|
| `NUXT_CACHE_INVALIDATION_KEY` | Shared API key expected by Nuxt |
| `NUXT_CACHE_INVALIDATION_SIGNING_SECRET` | HMAC signing secret. If omitted, Nuxt falls back to the API key |

## Request Bodies

### Product

```json
{
  "url": "/product/the-zen-platform-bed-47",
  "dryRun": false
}
```

`url` can also be a full URL. Only the pathname is used.

### Category

```json
{
  "url": "/category/51",
  "dryRun": false
}
```

### Homepage

```json
{
  "dryRun": false
}
```

The homepage endpoint always invalidates `/`; any `url` or `path` value is ignored.

## Response

```json
{
  "ok": true,
  "dryRun": false,
  "pageType": "product",
  "normalizedPath": "/product/the-zen-platform-bed-47",
  "pageKeys": ["..."],
  "dataKeys": ["..."],
  "deletedCount": 12
}
```

| Field | Description |
|---|---|
| `ok` | `true` when the request succeeded |
| `dryRun` | Whether the request only computed keys |
| `pageType` | `homepage`, `product`, `category`, or `unknown` |
| `normalizedPath` | Pathname used to build cache keys |
| `pageKeys` | Page cache keys deleted or computed |
| `dataKeys` | GraphQL query cache keys deleted or computed |
| `deletedCount` | Number of keys scheduled for deletion. `0` during dry-run |

## Page Cache Keys

Page cache keys are stored under:

```text
nitro:routes:_:
```

Some deployments can still expose page keys through older Nitro storage prefixes such as `/cache:pages:`. The invalidation code checks both formats.

The route cache plugin builds device-specific keys from:

```text
desktop-${path}
mobile-${path}
```

Nitro then removes non-word characters from that value. For example:

```text
desktop-/product/the-zen-platform-bed-47 -> desktopproductthezenplatformbed47
mobile-/product/the-zen-platform-bed-47  -> mobileproductthezenplatformbed47
```

In the Redis driver used locally, Nitro may shorten/hash the final page key, for example:

```text
nitro:routes:_:productthezenpla.blpGJSsBSZ.json
```

Because route names can be generated dynamically and page keys can be shortened, the invalidation code:

1. builds the known likely page keys
2. scans existing `nitro:routes` and legacy `pages` keys
3. deletes keys whose suffix, shortened slug prefix, or stored response body matches the requested URL

### Homepage

The homepage is `/`; it does not use a `/pages/...` URL pattern.

In the current production build, `nitro.prerender.routes` includes `/`, so the rendered homepage HTML can be served from `.output/public` and may not create a `nitro:routes:*` Redis entry at all. That is expected.

The homepage invalidation endpoint still removes the GraphQL data cache used by the home page:

```text
nitro:functions:_:GetWebsiteHomepageQuery-...
```

If a deployment stops prerendering `/` and lets the route pass through Nitro SWR, the same endpoint also attempts to remove homepage route cache keys for `/`.

## Data Cache Keys

GraphQL query cache keys are stored under:

```text
nitro:functions:_:
```

Some deployments can expose these as `/cache:nitro/functions:_:`. The invalidation code checks both prefixes.

The current query key shape is:

```text
{queryName}-{objectHash(variables)}-{isoCode}-{pricelist}.json
```

Storefront UI currently invalidates:

| Page type | Query keys |
|---|---|
| Product | `GetProductTemplateQuery` for slug variants, plus existing `GetProductVariantQuery` keys |
| Category | `GetCategoryQuery` for slug variants, plus existing `GetProductTemplateListQuery` keys |
| Homepage | `GetWebsiteHomepageQuery` |

Product/category slugs can be passed with or without a route prefix. For safety, the invalidation code checks these variants:

```text
/product/my-product
my-product
/my-product
```

and equivalent category variants.

## Dry Run

Send `"dryRun": true` to verify what would be deleted:

```json
{
  "url": "/product/the-zen-platform-bed-47",
  "dryRun": true
}
```

Dry-run computes `pageKeys` and `dataKeys` but does not call Redis deletion.

## When To Call

Call the invalidation endpoint immediately after Odoo changes content that affects the storefront.

| Odoo event | Endpoint |
|---|---|
| Product published/unpublished | `/api/internal/cache/invalidate/product` |
| Product name, description, image, SEO, price, or variant changed | `/api/internal/cache/invalidate/product` |
| Category name, SEO, hierarchy, or product membership changed | `/api/internal/cache/invalidate/category` |
| Homepage banner or content changed | `/api/internal/cache/invalidate/homepage` |

Use a short timeout from Odoo. Cache TTL remains a fallback if the invalidation request fails.
